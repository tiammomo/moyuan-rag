# RAG Interview QA

这份文档用于面试准备，聚焦业务难点、技术难点、落地成果和高频问答。

## 业务难点

### 1. 企业文档格式复杂，结构容易丢失

企业知识库里的文档通常来自 PDF、Word、Markdown、HTML、TXT 等多种格式，结构差异很大。如果解析和切片做得粗糙，标题、页码、段落边界会丢失，后续检索命中率会明显下降。

### 2. 用户问题和文档表述不一致

用户在提问时经常使用自然语言、简称、业务口语，而文档里可能是正式术语、制度编号、产品全称。只靠关键词检索很难稳定召回，只靠向量检索又容易召回语义相近但不够精确的内容。

### 3. 文档入库链路长，失败点多

上传之后还要经过解析、切片、向量化、ES 写入、Milvus 写入多个阶段。任一阶段失败，都可能导致文档卡在中间状态、重复入库或者知识库数据不一致。

### 4. 私有化部署和多组件运维复杂

这类系统通常要同时依赖 MySQL、Redis、Kafka、Elasticsearch、Milvus、前端、后端和多个 Worker。环境不一致时，很难稳定复现问题，也很难做联调和恢复。

## 技术难点与处理方式

### 1. 为什么 Kafka 不直接传全文和整批 chunks

直接传全文会带来几个问题：

- 消息体积大
- 重试成本高
- 容易触发大消息限制
- 失败回放不利于幂等

所以这里改成 Kafka 只传：

- `document_id`
- `file_path`
- `task_metadata`

全文和 chunks 改成落本地 artifact，再由下游按需读取。

### 2. 为什么要做混合检索

因为企业问答场景里，语义理解和关键词精确命中都很重要。

- 向量检索擅长语义相似
- 关键词检索擅长精确匹配

项目里把两者并行执行，再用 RRF 和后续 score 融合，兼顾召回率和准确率。

### 3. 怎么解决重复消费和重复入库

项目里主要做了三件事：

- consumer 关闭 auto-commit，只有处理成功或送入 DLQ 后才提交 offset
- worker 只允许合法状态进入当前阶段，防止陈旧消息重放污染状态
- vectorizer 在写入前先删除该文档已有的 ES / Milvus 数据，再执行重建

### 4. 为什么要引入 Alembic

直接在启动时 `create_all` 只适合非常早期的 Demo，不适合长期演进。项目里改成 Alembic 后：

- 数据结构变更可追踪
- 环境初始化更可控
- 部署与回滚流程更规范

### 5. 为什么还要做本地集成脚本

因为这不是单体应用，而是一套多组件系统。只看单元测试无法覆盖真正的链路问题，所以补了本地集成流程去校验：

- API 是否健康
- 上传到向量化是否跑通
- MySQL、ES、Milvus 是否一致落库

## 可以直接复用的面试回答

### Q1：你这个项目最大的难点是什么？

最大的难点不是单纯把大模型接起来，而是把文档处理、检索和运维做成完整工程体系。企业知识库场景下，文档格式复杂、问题表达多样、异步链路长、依赖组件多，所以我重点解决了三件事：异步链路可靠性、混合检索准确率和本地化部署可运维性。

### Q2：为什么选 Milvus + Elasticsearch，而不是只选一个？

因为两者解决的问题不一样。Milvus 更适合做语义召回，能处理“问法和原文不一样”的问题；Elasticsearch 更适合做关键词和短语匹配，能处理专有名词、制度编号、标题命中等场景。只用一个很难兼顾召回率和精确度，所以项目里采用并行混合召回。

### Q3：你在可靠性上做了什么？

我把 Kafka 消费改成手动提交 offset，并加了 DLQ、回放、阶段幂等和失败闭环。这样消息处理失败不会直接丢失，重复消费也不会导致文档状态错乱或向量重复入库。

### Q4：怎么证明这套链路是真的可用？

项目里补了本地集成脚本和 compose 运维脚本，能从启动环境、执行迁移、上传文档一直验证到 ES 和 Milvus 落库。我会用 smoke test 去验证完整链路，而不是只看单个接口是否返回 200。

### Q5：这项目更偏 AI 还是更偏工程？

我会说它是 AI 应用工程。AI 侧体现在 RAG、混合检索和问答生成；工程侧体现在 Kafka 流水线、迁移治理、容器编排、DLQ、幂等和集成验证。真正有价值的是把两边结合起来，而不是只做模型调用 Demo。

## 面试时推荐强调的落地成果

- 完整 RAG 链路已经跑通，不是纸面设计
- 文档处理由同步调用演进为可恢复的异步流水线
- 检索从单一路径演进为 Milvus + Elasticsearch 混合召回
- 运维从手动拼环境演进为统一 Compose 编排和脚本化操作
- 数据结构从启动建表演进为 Alembic 迁移体系

## 适合最后总结的一句话

这个项目最有价值的地方，是把 RAG 从“能演示”推进到了“能联调、能恢复、能部署、能运维”的工程阶段。
