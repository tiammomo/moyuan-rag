# RAG Workflow And Hybrid Retrieval

这份文档用于项目汇报、面试讲解和对外介绍，重点说明当前项目里 RAG 的完整工作流，以及混合检索是如何落地的。

## 整体链路

系统可以拆成两条主线：

1. 离线入库链路：`文件 -> 解析 -> 切片 -> 向量化 -> 写入 ES / Milvus`
2. 在线问答链路：`问题 -> 混合召回 -> 融合 / 重排 -> 生成答案`

## 离线入库 Workflow

### 1. 用户上传文档

- 用户把 PDF、DOCX、TXT、Markdown、HTML 等文档上传到知识库
- 系统先保存原始文件和文档元数据
- 文档状态初始化为 `uploading`
- Kafka 只发送轻量任务消息，不直接传全文内容

消息模型重点是：

- `document_id`
- `file_path`
- `task_metadata`

这样做的好处是：

- 降低 Kafka 消息体积
- 减少大消息传输风险
- 更利于失败重试和幂等控制

### 2. Parser 阶段

- `parser worker` 消费上传消息
- 读取原始文件并做格式解析
- 输出标准化文本内容
- 中间产物落地到本地 pipeline artifact，而不是继续把全文塞进 Kafka
- 文档状态更新为 `parsing`

### 3. Splitter 阶段

- `splitter worker` 读取解析结果
- 按知识库配置的 `chunk_size / chunk_overlap` 做切片
- 保留标题、页码等结构信息
- 输出切片中间产物
- 文档状态更新为 `splitting`

### 4. Vectorizer 阶段

- `vectorizer worker` 读取切片结果
- 生成每个 chunk 的 embedding
- 向量写入 `Milvus`
- chunk 正文和结构化信息写入 `Elasticsearch`
- 文档状态更新为 `embedding`

### 5. 完成与清理

- 所有写入成功后，文档状态更新为 `completed`
- 同步更新知识库的文档数量和 chunk 数量
- 清理中间 pipeline artifact
- 如果任何阶段失败，则状态转为 `failed`

## 为什么要拆成三段 Worker

这套设计主要是为了解决企业知识库场景里的几个现实问题：

- 文档处理链路长，单点失败概率高
- PDF / Word 解析和向量化耗时差异很大
- 不同阶段的 CPU / IO / 网络瓶颈不同
- 后续需要支持重试、回放、死信和幂等治理

拆成 `parse / split / vectorize` 后：

- 每个阶段职责更清晰
- 故障定位更容易
- 支持单独扩容高负载阶段
- 更适合接入 DLQ 和重试机制

## 在线问答 Workflow

### 1. 用户发起问题

- 用户选择机器人并输入问题
- 系统读取机器人绑定的知识库集合
- 加载聊天历史和检索配置

### 2. 查询准备

- 根据当前问题和近期上下文生成检索查询
- 如果是追问场景，会把最近轮次的上下文纳入检索输入

### 3. 并行混合召回

系统同时执行两路召回：

- 向量召回：去 `Milvus` 做语义检索
- 关键词召回：去 `Elasticsearch` 做全文 / 短语匹配

### 4. 结果融合

项目里的融合思路是：

- 先拿到两路候选结果
- 用 `RRF` 做排序融合
- 再生成最终 blended score
- 必要时再走 rerank

这样做的目的是在“语义相似”和“关键词精确命中”之间取得平衡。

### 5. 上下文构建与生成

- 取融合后的 TopK chunk
- 拼接成知识上下文
- 与机器人 prompt、历史消息一起送给大模型
- 返回最终答案和引用来源

## 混合检索为什么重要

如果只做向量检索，容易出现：

- 语义相近但事实不精确
- 专有名词、制度编号、产品名称命中不稳定

如果只做关键词检索，容易出现：

- 表述方式变化后召回率下降
- 用户自然语言问法和文档写法不一致时命中差

所以这个项目选的是混合方案：

- `Milvus` 负责“像不像”
- `Elasticsearch` 负责“是不是同一个词 / 短语 / 标题”

## 当前项目里的工程化重点

- Kafka 只传轻量消息，不传大文本
- worker 只在成功或进入 DLQ 后提交 offset
- 支持 DLQ 查看与回放
- 向量化阶段做先删后写，避免重复消费导致重复索引
- 通过本地集成脚本验证 MySQL、ES、Milvus 三端落库一致性

## 对外讲解时的精简版本

这个项目的 RAG 不是单路检索，而是完整的“异步文档处理 + 混合检索 + 生成答案”体系。离线侧用 Kafka 把文档处理拆成解析、切片、向量化三段流水线，在线侧并行执行 Milvus 语义召回和 Elasticsearch 关键词召回，再做融合和重排，把最相关的 chunk 交给大模型生成最终答案。
