# 04 生成答案

## 这一步发生在什么时候

当混合检索拿到候选片段，并完成融合、重排、过滤后，系统才会进入真正的生成阶段。

也就是说，大模型不是直接回答用户问题，而是“基于检索上下文回答”。

## 生成前要准备什么

系统会先收集这些内容：

- 当前用户问题
- 最近会话历史
- 机器人 `system_prompt`
- 检索出来的 topK 上下文

## 上下文怎么组装

项目里的做法是把召回片段整理成可引用的上下文块，然后和用户问题一起交给 LLM。

通常会包含：

- chunk 正文
- 来源信息
- 排序后的相关片段列表

## 为什么要把检索结果显式拼进 prompt

因为这样能让回答更可控：

- 减少模型凭空编造
- 让答案尽量基于已知知识库内容
- 更容易解释“答案来自哪里”

## 会话历史的作用

如果用户不是第一次提问，会话历史有两个作用：

1. 帮助检索理解“这次是在追问什么”
2. 帮助生成阶段保持上下文连续

## 生成阶段真正做的事

可以粗略理解成：

```text
用户问题
+ 检索上下文
+ system prompt
+ 历史消息
= 最终给大模型的输入
```

## 为什么这一步不能脱离检索独立看

如果只看生成，很容易误以为“答案质量主要取决于大模型”。

但在 RAG 里，答案质量很大程度上取决于：

- 检索拿到的上下文对不对
- 上下文有没有被噪声污染
- topK 是否合理

所以生成质量本质上是“检索质量 + prompt 组织 + 模型能力”的组合结果。

## 这一章的重点

RAG 不是“先查一下再问模型”，而是“先把相关上下文组织好，再让模型在这个上下文里回答”。
