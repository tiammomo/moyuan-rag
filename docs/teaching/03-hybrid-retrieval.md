# 03 混合检索

## 为什么要做混合检索

如果只做向量检索，容易出现：

- 语义相关，但细节不够精确
- 专有名词、制度标题、缩写命中不稳定

如果只做关键词检索，容易出现：

- 用户问法变化后命中率下降
- 自然语言问题和文档正式表述不一致时效果差

所以这个项目采用并行的混合检索。

## 这个项目里的两路召回

### 向量召回

由 `Milvus` 负责。

它解决的是：

- 用户问题和原文表达不同
- 需要按语义相似度找相关片段

### 关键词召回

由 `Elasticsearch` 负责。

它解决的是：

- 产品名、制度名、专有术语
- 标题命中
- 短语精确匹配

## 在线查询流程

```text
问题
-> 查询准备
-> 向量召回 + 关键词召回并行执行
-> 结果融合
-> 可选 rerank
-> 阈值过滤
-> 组装上下文
-> 交给大模型
```

## 第 1 步：查询准备

系统会先加载：

- 机器人配置
- 绑定的知识库
- 最近会话上下文

如果是追问场景，还会把最近一轮上下文拼进 retrieval query，帮助检索命中。

## 第 2 步：并行召回

这一步不是先查向量再查关键词，而是同时执行两条路径：

- 向量检索去 Milvus
- 关键词检索去 Elasticsearch

这样可以减少等待时间，也让后续融合有完整候选集。

## 第 3 步：结果融合

当前项目里融合分两层理解：

### 1. 排序层

先用 `RRF` 做融合排序。

RRF 的作用很简单：

- 一个 chunk 如果两边都命中，排名会更稳定
- 避免只依赖单一检索分数

### 2. 分数层

融合后再生成最终的 blended score，用来表示综合相关度。

## 第 4 步：可选 rerank

如果机器人启用了 rerank，系统会在融合候选集上再做一次精排。

它的作用是：

- 把初筛后的候选重新比较
- 提升最终 topK 的质量

## 第 5 步：阈值过滤

在把上下文交给大模型之前，还会按 `similarity_threshold` 做过滤。

目的是：

- 把明显不相关的片段挡掉
- 避免垃圾上下文干扰生成

## 为什么企业场景更需要混合检索

企业知识问答常见这些内容：

- 制度标题
- 产品编号
- 项目名称
- 术语缩写
- 固定模板措辞

这些内容既需要“语义理解”，也需要“精确命中”。所以单路检索通常不够。

## 这一章的重点

混合检索的关键不只是“两边都查”，而是：

- 并行召回
- 统一候选集
- 融合排序
- 可选精排
- 最后再过滤
