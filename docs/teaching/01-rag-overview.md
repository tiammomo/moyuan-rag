# 01 RAG 总览

## 什么是这个项目里的 RAG

在这个项目里，RAG 不是“把问题丢给大模型，再顺便查一下资料”，而是一套完整链路：

1. 离线入库链路：文档上传后，经过解析、切片、向量化，最终写入 Elasticsearch 和 Milvus。
2. 在线问答链路：用户提问后，先做检索，再把检索到的上下文交给大模型生成答案。

## 两条主线

### 1. 离线入库

```text
文件 -> 解析 -> 切片 -> 向量化 -> 写入 ES / Milvus
```

### 2. 在线问答

```text
问题 -> 查询准备 -> 混合召回 -> 融合 / 重排 -> 生成答案
```

## 为什么不能只做一个简单 Demo

如果只是简单 Demo，通常会这样做：

- 文档直接读进内存
- 文本一次性切片
- 全部向量写入一个库
- 提问时只做一次向量搜索

这样实现快，但很难应对真实项目里的问题：

- 文档格式复杂，解析容易出错
- 入库链路长，任何阶段都可能失败
- 企业场景里专有名词很多，单纯向量检索不稳定
- 多个服务一起跑时，调试和恢复都很麻烦

## 这个项目的核心设计

- 文档处理拆成 `parser / splitter / vectorizer` 三段 worker
- Kafka 只传轻量任务指针，不传全文和整批 chunks
- Milvus 负责语义召回，Elasticsearch 负责全文和关键词召回
- 检索不是单路，而是并行的混合检索
- 本地运行不是“各自手动启动”，而是统一 compose 和运维脚本

## 一句话理解整个系统

这个项目的 RAG，本质上是“用异步流水线把文档变成可检索知识，再用混合检索把最相关的内容交给大模型生成答案”。
