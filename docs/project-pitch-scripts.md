# Project Pitch Scripts

这份文档给出 30 秒、60 秒和 3 分钟三个版本的项目介绍稿，适合不同面试场景直接复用。

## 30 秒版本

我做的是一个企业级 RAG 知识问答系统，支持多格式文档上传、异步解析切片、向量化入库和混合检索问答。项目里我重点解决了文档处理链路可靠性、Milvus 加 Elasticsearch 的混合检索准确率，以及多组件本地化部署和运维问题，把它从一个 AI Demo 做成了可联调和可恢复的工程系统。

## 60 秒版本

这个项目是一个面向企业知识库场景的 RAG 平台。离线侧，文档会经过上传、解析、切片、向量化三个异步阶段，最终把向量写到 Milvus，把全文和 chunk 内容写到 Elasticsearch。在线侧，系统会并行执行向量召回和关键词召回，再做融合和必要的重排，把最相关的上下文交给大模型生成答案。  
我在里面重点做了 Kafka 异步流水线、消息轻量化改造、DLQ 和幂等治理、Alembic 迁移体系、以及 Compose 统一编排和集成验证，让这套系统不仅能跑，而且能稳定联调和恢复。

## 3 分钟版本

我做的是一个企业级 RAG 知识问答系统，目标是面向企业内部知识管理、制度查询、产品文档问答这类场景，提供一个可以本地部署的知识问答平台。  

这个项目我觉得最重要的点，不是单纯接一个大模型，而是把整个 RAG 链路工程化。离线入库侧，文档上传以后不会同步一路处理到底，而是拆成 `parse / split / vectorize` 三段 Kafka Worker 流水线。这样做有几个好处：第一，职责更清晰；第二，哪一段慢就单独扩；第三，更容易做失败重试、死信和回放。  

我还把 Kafka 的消息模型做了调整，改成只传 `document_id / file_path / task metadata` 这种轻量消息，不再传全文和整批 chunk。全文和切片结果会作为中间产物落盘，由后续 worker 按需读取。这个改动主要是为了解决大消息、重复消费和重放成本的问题。  

在线检索侧，我没有只做向量检索，而是用了 Milvus 和 Elasticsearch 的混合检索。因为企业场景里很多问题依赖专有名词、制度编号、产品名称这类精确匹配，如果只靠向量检索，很容易召回“语义相近但不够准”的结果。所以系统会并行跑语义召回和关键词召回，再做 RRF 融合、阈值过滤，必要时再 rerank。这样可以兼顾召回率和精确度。  

除了检索逻辑，我还补了很多工程能力。比如 Kafka 消费改成手动提交 offset，只在处理成功或者进入 DLQ 以后才提交；vectorizer 会先删后写，避免重复消费导致重复索引；数据库也从启动期 `create_all` 改成 Alembic 迁移体系。  

最后一块是部署和验证。我把前后端、MySQL、Redis、Kafka、Elasticsearch、Milvus 全部纳入 Docker Compose，补了启动、状态、日志、重启、停机和本地集成脚本，保证可以从环境拉起一直验证到文档上传、ES 落库和 Milvus 落库。  

所以如果要概括这个项目，我会说它最大的价值，是把 RAG 从一个“能跑的 AI Demo”，推进成了一个“能联调、能恢复、能部署、能运维”的工程系统。

## 面试时的补充技巧

- 面试官时间少时，用 30 秒版
- 面试官追问技术细节时，用 60 秒版
- 面试官让完整介绍项目时，用 3 分钟版
- 如果对方更偏工程，就重点讲 Kafka、DLQ、迁移、Compose
- 如果对方更偏 AI，就重点讲混合检索、RAG workflow、召回和重排
